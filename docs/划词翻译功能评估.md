# 划词翻译功能评估

## 功能目标

在现有剪贴板翻译工具的基础上增加一个“划词翻译模式”。用户开启该模式后，在其他应用中选中文本，应用在选中文本附近或鼠标附近弹出一个小窗口，显示翻译结果。

这个功能比图片翻译更适合优先做，因为它和 CopyTranslator 类工具的定位非常契合，工程复杂度也比 OCR 图片翻译低很多。

第一版建议只支持 Google 和百度翻译引擎。划词翻译通常需要较低延迟，Google 和百度对短文本、词组、句子的响应速度和稳定性更适合。LLM API 和 Ollama 可以后续作为可选项，但不建议作为第一版默认方案。

## Tauri 是否能做弹窗

Tauri 可以实现这类弹窗。推荐使用一个独立的透明置顶小窗口，例如：

```text
selection-popup
  decorations: false
  transparent: true
  always_on_top: true
  resizable: false
  skip_taskbar: true
```

弹窗可以由独立前端页面渲染，例如 `selection-popup.html`，用于显示翻译结果。这样不会影响当前主窗口，也更容易单独控制位置、大小、关闭和样式。

弹窗位置可以使用鼠标坐标决定：

```text
选中文本后触发翻译
  -> 获取鼠标当前位置
  -> 翻译完成
  -> 将 selection-popup 移动到鼠标下方
  -> 显示译文
```

如果要做到“严格显示在文本下方”，难度会高很多，因为跨应用获取选区矩形不是普通桌面应用天然能做到的。第一版建议显示在鼠标附近，用户体验已经接近划词翻译。

## 最大难点：获取选中文本

Tauri 应用无法天然读取其他应用中的选中文本。比如用户在浏览器、PDF 阅读器、Word、微信、IDE 中选中文本，Tauri 本身并不能直接拿到这段文字。

因此必须选择一种触发和取词方案。

## 方案一：快捷键触发

流程：

```text
用户选中文本
  -> 按快捷键，例如 Alt+Q
  -> 应用读取当前剪贴板或主动执行复制逻辑
  -> 调用 Google/百度翻译
  -> 在鼠标附近显示弹窗
```

这是第一版最推荐的方案。

优点：

- 稳定。
- 用户意图明确。
- 不需要持续监听鼠标选区。
- 不容易误触发。
- 对不同应用兼容性最好。

缺点：

- 不是完全自动的“选中即翻译”。
- 如果不模拟复制，应用仍然拿不到其他应用里的选中文本。

## 方案二：模拟 Ctrl+C

流程：

```text
用户选中文本
  -> 应用检测鼠标抬起或快捷键
  -> 保存原剪贴板内容
  -> 模拟 Ctrl+C
  -> 读取剪贴板文本
  -> 恢复原剪贴板内容
  -> 翻译并显示弹窗
```

这是很多划词翻译工具的实际做法。

优点：

- 能覆盖大多数普通应用。
- 可以实现更接近“选中后自动翻译”的体验。
- 不依赖目标应用提供 API。

缺点：

- 会短暂改动用户剪贴板。
- 有些应用复制较慢，需要等待和重试。
- 有些应用禁止复制或复制出来是富文本。
- 可能和用户手动复制操作冲突。
- 需要谨慎保存和恢复剪贴板内容。

如果采用这个方案，必须设计 ClipboardGuard：

```text
保存原剪贴板
模拟 Ctrl+C
读取新文本
如果文本有效则翻译
恢复原剪贴板
```

## 方案三：Windows 辅助功能 API / UI Automation

理论上可以通过 Windows UI Automation 获取当前焦点控件、选中文本或文本范围。

优点：

- 不一定需要污染剪贴板。
- 对部分标准控件可以直接拿到选中文本。

缺点：

- 跨应用兼容性不稳定。
- 浏览器、PDF、Electron 应用、游戏、部分自绘 UI 不一定支持。
- 开发复杂度明显高于模拟 Ctrl+C。
- 需要处理 Windows 平台特定逻辑，跨平台成本高。

这个方案可以作为后续增强，但不建议第一版依赖它。

## 方案四：OCR 鼠标附近文本

也可以截取鼠标附近区域，用 OCR 识别文本。

优点：

- 不依赖目标应用是否允许复制。
- 对图片、PDF 扫描件也可能有效。

缺点：

- 复杂度接近图片翻译。
- 速度和准确率不稳定。
- 很难准确知道用户真正选中的文本范围。

不建议作为划词翻译第一版方案。

## 推荐触发策略

建议提供两种模式：

```text
1. 快捷键模式，默认推荐
   用户选中文本后按快捷键翻译。
   稳定、权限少、误触发低。

2. 自动划词模式，高级选项
   鼠标选中文本后自动尝试复制并翻译。
   体验更像词典，但可能影响剪贴板。
```

第一版可以先实现快捷键模式。后续再加自动划词模式。

## 推荐模块划分

```text
SelectionModeSettings
  -> 保存划词翻译开关、触发方式、引擎、快捷键等设置

SelectionTrigger
  -> 监听快捷键或鼠标事件

SelectionTextProvider
  -> 获取选中文本
  -> 第一版可以使用剪贴板/模拟复制
  -> 后续可扩展 UI Automation

SelectionTranslator
  -> 调用 Google 或百度翻译

SelectionPopupWindow
  -> 控制弹窗创建、定位、显示和隐藏

ClipboardGuard
  -> 保存和恢复剪贴板，避免破坏用户剪贴板
```

## 设置项建议

```text
selection_translation_enabled: boolean
selection_trigger_mode: "hotkey" | "auto"
selection_engine: "google" | "baidu"
selection_hotkey: string
selection_popup_auto_close_ms: number
selection_restore_clipboard: boolean
```

设置界面可以放在“通用”或新增“划词翻译”区域。

## 弹窗交互建议

弹窗应尽量轻量：

- 默认显示在鼠标下方。
- 如果超出屏幕边缘，自动向上或向左偏移。
- 失焦或鼠标移开后自动关闭。
- 支持复制译文。
- 支持点击固定弹窗，避免自动消失。
- 显示翻译中状态，避免用户以为没有响应。

弹窗内容可以包含：

```text
原文短预览
译文
当前引擎
复制按钮
关闭按钮
```

## 实现风险

主要风险在“取词”，不是弹窗。

需要注意：

- 不同应用复制行为不一致。
- 模拟 Ctrl+C 可能影响用户剪贴板。
- 自动划词可能误触发。
- 短时间频繁触发需要防抖。
- 翻译请求需要超时控制。
- 弹窗必须避免抢焦点，否则可能影响用户继续操作。

## 推荐落地路线

第一阶段：

```text
划词翻译开关
快捷键触发
读取剪贴板文本
Google/百度翻译
鼠标附近 Tauri 弹窗显示
```

第二阶段：

```text
模拟 Ctrl+C 获取选中文本
保存并恢复剪贴板
弹窗自动关闭和固定
异常提示
```

第三阶段：

```text
自动划词模式
鼠标抬起后防抖触发
更精确的弹窗定位
可选 UI Automation 取词
```

## 结论

划词翻译功能可行，且比图片翻译更适合优先实现。Tauri 可以做透明置顶弹窗，也可以将弹窗放在鼠标附近显示。

第一版最稳的方案是：

```text
用户选中文本
  -> 按快捷键
  -> 应用获取文本
  -> 调用 Google/百度
  -> 鼠标附近显示翻译弹窗
```

如果要做到“选中后自动翻译”，大概率仍然需要模拟 Ctrl+C 或使用 Windows UI Automation。综合兼容性和开发成本，建议先做快捷键模式，再逐步增强为自动划词模式。
# 划词翻译功能评估

## 背景

计划在当前翻译工具中增加一个“划词翻译模式”。用户开启该模式后，在其他应用中选中文本，应用在选中文本附近弹出一个小窗口显示翻译结果。

这个功能和当前项目定位非常契合。相比图片翻译，划词翻译的工程复杂度更低、反馈更快，也更接近 CopyTranslator 类工具的核心体验。

第一版建议只支持 Google 和百度翻译引擎。划词翻译通常面向短句、词组和段落片段，要求响应快、结果稳定。Google 和百度比 LLM/Ollama 更适合默认承担这个场景。

## Tauri 是否能做到文本下方弹窗

Tauri 可以做到。

推荐做一个独立的透明悬浮窗口，例如：

```text
label: selection-popup
decorations: false
transparent: true
always_on_top: true
resizable: false
skip_taskbar: true
focus: false 或尽量避免抢焦点
```

工作流程：

```text
获取选中文本
  -> 获取鼠标位置或选区附近位置
  -> 调用翻译引擎
  -> 移动 selection-popup 到目标位置
  -> 显示翻译结果
  -> 鼠标离开、失焦或超时后隐藏
```

需要注意的是，Tauri 可以创建并移动窗口，但“准确拿到其他应用中文本选区的屏幕坐标”并不总是容易。更现实的第一版做法是把弹窗显示在鼠标附近，而不是严格贴在选中文本正下方。

后续如果要更精确贴近文本下方，可以在 Windows 上探索 UI Automation 获取选区 bounding rectangle，但兼容性和工作量会明显增加。

## 获取选中文本的方式

### 方案一：快捷键触发并读取剪贴板

用户选中文本后，按一个全局快捷键，例如：

```text
Alt + Q
Ctrl + C + C
```

应用读取剪贴板内容，调用翻译，然后在鼠标附近显示弹窗。

优点：

- 实现简单。
- 稳定性较好。
- 不需要持续监听鼠标选区。
- 用户明确触发，不容易误翻译。

缺点：

- 用户需要多按一次快捷键。
- 如果快捷键前没有复制文本，需要应用主动触发复制，或者要求用户先复制。

这是最推荐的第一版方案。

### 方案二：鼠标选中后模拟 Ctrl+C

开启划词翻译模式后，应用监听鼠标抬起事件。检测到用户完成选区后，应用自动模拟 `Ctrl+C`，读取剪贴板文本，翻译并弹窗显示。

工作流程：

```text
监听鼠标按下
  -> 监听鼠标拖拽
  -> 监听鼠标抬起
  -> 短暂等待选区稳定
  -> 保存原剪贴板
  -> 模拟 Ctrl+C
  -> 读取新剪贴板文本
  -> 恢复原剪贴板
  -> 翻译并显示弹窗
```

优点：

- 体验接近真正的“划词即翻译”。
- 不要求用户手动复制。

缺点：

- 会短暂改动用户剪贴板。
- 在某些应用中可能复制失败。
- 可能和用户正在进行的复制操作冲突。
- 有些软件禁止复制或复制结果是富文本/空文本。
- 需要全局鼠标和键盘模拟能力，Windows 权限与兼容性要仔细处理。

如果做这个方案，必须加 `ClipboardGuard`，尽量保存并恢复用户原剪贴板内容。

### 方案三：Windows UI Automation 获取选中文本

理论上可以不模拟 `Ctrl+C`，而是通过 Windows UI Automation 读取当前焦点控件的选中文本。

大致思路：

```text
获取当前前台窗口
  -> 获取焦点控件
  -> 查询 TextPattern / SelectionPattern
  -> 读取 selected text
  -> 尝试读取选区 bounding rectangle
```

优点：

- 不污染剪贴板。
- 有机会拿到选区坐标，方便把弹窗贴近文本下方。

缺点：

- 跨应用兼容性不稳定。
- 浏览器、PDF、Electron 应用、游戏窗口、自绘 UI 表现不一致。
- 有些应用不暴露 TextPattern。
- Rust 侧实现复杂度高。
- 权限、焦点、DPI 缩放、多显示器都要处理。

这个方案适合后续作为高级优化，不适合第一版。

### 方案四：应用内划词翻译

如果只要求在本应用自己的窗口里选中文本，可以直接用前端 Selection API：

```ts
window.getSelection()
```

优点是简单、准确、能拿到选区坐标。缺点是只能覆盖应用内文本，无法对浏览器、PDF、Word、IDE 等外部应用生效。

对于这个项目来说，应用内划词意义有限，因为主要需求是翻译其他应用里的文本。

## 不模拟 Ctrl+C 的可行替代方案

如果完全不模拟 `Ctrl+C`，可选方案主要有：

1. **全局快捷键 + 读取当前剪贴板**
   用户自己复制文本后按快捷键翻译。最稳定，但不是严格划词即翻译。

2. **Windows UI Automation**
   理论上可以直接读取选区文本和选区位置，但兼容性和实现复杂度高。

3. **OCR 识别选区截图**
   鼠标选中后截取附近区域，用 OCR 识别文本。可以绕过复制限制，但复杂度更高，且准确性不如直接复制文本。

4. **浏览器扩展/插件**
   对网页划词体验最好，但只覆盖浏览器，不适合作为桌面全局方案。

综合看，桌面端全局划词翻译如果想兼容大多数软件，模拟复制仍然是最现实的方案。

## 推荐第一版实现

第一版建议做“快捷键触发模式”，而不是直接做自动鼠标划词。

推荐流程：

```text
用户开启划词翻译模式
  -> 用户在任意应用选中文本
  -> 用户按全局快捷键
  -> 应用模拟 Ctrl+C 或读取剪贴板
  -> 调用 Google / 百度翻译
  -> 在鼠标附近显示小弹窗
```

如果希望更接近自动划词，可以在设置里提供实验选项：

```text
触发方式：
1. 快捷键触发，默认推荐
2. 鼠标选中后自动触发，实验功能
```

## 建议的模块拆分

```text
SelectionModeSettings
  -> 是否开启划词翻译
  -> 触发方式
  -> 使用 Google / 百度
  -> 弹窗自动关闭时间

SelectionMonitor
  -> 监听全局快捷键
  -> 可选监听鼠标拖拽和抬起

ClipboardGuard
  -> 保存原剪贴板
  -> 模拟复制
  -> 读取文本
  -> 恢复原剪贴板

SelectionTranslator
  -> 调用 Google / 百度翻译
  -> 处理超时和错误

SelectionPopupWindow
  -> 透明悬浮窗口
  -> 显示译文
  -> 移动到鼠标附近或选区附近
  -> 自动隐藏
```

## 窗口位置建议

第一版建议把弹窗放在鼠标附近：

```text
left = mouse_x
top = mouse_y + 12
```

并处理边界：

```text
如果右侧超出屏幕，则向左显示
如果下方超出屏幕，则显示在鼠标上方
```

这样比“严格贴在选中文本正下方”更容易实现，也足够接近用户预期。

如果后续接入 UI Automation 并能拿到选区矩形，再改成：

```text
left = selection_rect.left
top = selection_rect.bottom + 6
```

## 风险点

- 全局鼠标/键盘监听在 Windows 上需要额外实现或插件。
- 模拟 `Ctrl+C` 可能影响用户剪贴板，需要保存和恢复。
- 某些软件复制选中文本失败。
- 某些应用复制结果是富文本或带格式内容，需要提取纯文本。
- 弹窗不能抢焦点，否则会影响用户原应用操作。
- 多显示器和 DPI 缩放下窗口定位要特别测试。
- 翻译请求需要短超时，否则弹窗等待体验差。

## 结论

划词翻译功能可行，而且建议优先级高于图片翻译。

Tauri 可以实现透明悬浮弹窗，并把翻译结果显示在鼠标附近或选区附近。第一版推荐使用全局快捷键触发，搭配 Google / 百度翻译引擎，先做稳定体验。自动鼠标划词和 Windows UI Automation 可以作为后续增强。

推荐路线：

```text
第一阶段：快捷键触发 + 鼠标附近弹窗
第二阶段：自动模拟 Ctrl+C + 剪贴板保护
第三阶段：Windows UI Automation 获取选区文本和坐标
```
